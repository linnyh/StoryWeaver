# StoryWeaver 分步改进计划

本文档将改进项分为四个阶段：**收尾与修复**、**体验与可见性**、**能力增强**、**进阶功能**。建议按顺序执行，每阶段完成后再进入下一阶段。

---

## 阶段一：收尾与修复（优先完成）

目标：修复明显问题、统一配置与规范，为后续功能打基础。

### 1.1 仓库与配置收尾

| 步骤 | 内容 | 负责模块 | 验收标准 |
|------|------|----------|----------|
| 1.1.1 ✅ | 将 `*.db-wal`、`*.db-shm` 加入 `.gitignore`，避免 SQLite WAL 文件被提交 | 根目录 | `git status` 不再出现 db-wal/db-shm |
| 1.1.2 ✅ | 补充/完善 `backend/.env.example`：列出 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`OPENAI_MODEL` 等必填项及说明 | backend | 新克隆项目可按 example 配置并启动 |
| 1.1.3 ✅ | 生产环境 CORS：从 `allow_origins` 中移除 `"*"`，仅保留明确前端域名 | backend `main.py` | 生产部署时不会对任意域名开放 |

### 1.2 API 与前端请求一致性

| 步骤 | 内容 | 负责模块 | 验收标准 |
|------|------|----------|----------|
| 1.2.1 ✅ | 确认 Vite proxy：`/api` → 后端服务（如 `http://localhost:8000`） | frontend `vite.config.js` | 前端请求 `/api/xxx` 能正确转发到后端 |
| 1.2.2 ✅ | 统一前端 API baseURL：若 proxy 已转发 `/api`，则 baseURL 用 `''` 或 `'/api'` 且所有请求路径不再重复写 `/api` | frontend `src/api/index.js` | 无 404、无重复路径 |

### 1.3 RAG 按小说隔离

| 步骤 | 内容 | 负责模块 | 验收标准 |
|------|------|----------|----------|
| 1.3.1 ✅ | 在所有 RAG 检索入口（如 `retrieve_context`、写作上下文组装）中传入并过滤 `novel_id`，确保只检索当前小说的角色/设定/摘要 | backend `app/rag/service.py`、调用 RAG 的 usecase/generator | 切换小说时不会混用其他小说的 RAG 内容 |
| 1.3.2 ✅ | 为 RAG 检索增加单测：给定不同 `novel_id` 的数据，验证检索结果仅来自对应小说 | backend tests | 有覆盖 novel 隔离的用例 |

## 阶段二：体验与可见性

目标：让首页数据真实、导航清晰、核心能力（关系图谱、审稿日志）易于发现。

### 2.1 首页与导航

| 步骤 | 内容 | 负责模块 | 验收标准 |
|------|------|----------|----------|
| 2.1.1 ✅ | 新增统计接口：如 `GET /api/novels/stats`，返回当前（或全局）的 novels 数、总章节数、总字数 | backend `app/api/novel.py` | 接口返回正确聚合数据 |
| 2.1.2 ✅ | 首页使用上述接口展示「Novels / Chapters / Words」真实数据，去掉写死的 12、24k | frontend `HomePage.vue` | 数字随数据变化 |
| 2.1.3 ✅ | 侧栏导航：若「首页」与「我的作品」功能重复，合并为一个入口或区分（如「工作台」+「作品库」），并修正 `router-link` 的 `to` | frontend `App.vue` | 无重复入口、点击行为符合预期 |

### 2.2 关系图谱与审稿日志入口

| 步骤 | 内容 | 负责模块 | 验收标准 |
|------|------|----------|----------|
| 2.2.1 ✅ | 在小说详情页（NovelPage）或侧栏增加「关系图谱」入口，跳转至关系图谱视图（或已有 RagPage/独立路由） | frontend | 从小说页可一键打开关系图谱 |
| 2.2.2 ✅ | 写作页「Logs」在无日志时显示简短说明：「开启 Review 并生成后将在此显示审稿日志」 | frontend `WriterPage.vue` | 新用户能理解 Logs 用途 |

### 2.3 文档与帮助

| 步骤 | 内容 | 负责模块 | 验收标准 |
|------|------|----------|----------|
| 2.3.1 ✅ | 首页「Documentation」按钮：链接到 README、DEV_DOC 或在线文档；若无则改为「帮助」并链到项目内说明 | frontend `HomePage.vue` | 按钮有效、不误导 |

## 阶段三：能力增强

目标：提升写作助手可用性、导出与备份、基础版本能力。

### 3.1 场景 Chat 与写作助手

| 步骤 | 内容 | 负责模块 | 验收标准 |
|------|------|----------|----------|
| 3.1.1 | 写作页右侧助手区域增加简短说明或预设话术（如「把这段改得更悲伤」「在这里加一段环境描写」） | frontend `WriterPage.vue` | 用户能快速理解可如何与 AI 对话 |
| 3.1.2 | 场景 Chat 上下文增强：除 `scene.content` 外，可选带入 `beat_description`、上一场景 summary 或少量 RAG 设定 | backend `scene_chat.py`、generator/rag | 改写建议更贴设定与前后文 |

### 3.2 导出与备份

| 步骤 | 内容 | 负责模块 | 验收标准 |
|------|------|----------|----------|
| 3.2.1 | 在 README 或 UI 中明确说明当前导出格式（如 PDF/DOCX/Markdown）及导出范围（全书/当前章） | docs + frontend | 用户知道能导出什么、什么格式 |
| 3.2.2 | 可选：支持「导出设定集」（角色+世界观）为单独文件，便于备份或迁移 | backend + frontend | 有入口且导出内容正确 |

### 3.3 审稿开关与默认值

| 步骤 | 内容 | 负责模块 | 验收标准 |
|------|------|----------|----------|
| 3.3.1 | 根据产品偏好决定审稿默认值：若优先速度则 `enable_editorial` 默认 `False`，并在 UI 标注「开启后生成更慢但会进行逻辑/爽点/思想审查」 | backend 默认参数 + frontend 文案 | 用户理解开关含义与代价 |

## 阶段四：进阶功能

目标：版本历史、多模型配置、体验细节与测试完善。

### 4.1 场景/章节版本历史

| 步骤 | 内容 | 负责模块 | 验收标准 |
|------|------|----------|----------|
| 4.1.1 | 设计并实现场景版本表（如 `scene_id, version, content, created_at`），在保存场景内容时写入新版本（可设上限如最近 20 版） | backend models + scene usecases | 每次保存产生可追溯版本 |
| 4.1.2 | 提供「历史版本」列表接口与「恢复此版本」接口 | backend API | 可列出并恢复指定版本 |
| 4.1.3 | 写作页增加「历史版本」入口：列表 + 预览 + 恢复 | frontend `WriterPage.vue` | 用户能查看并回滚 |

### 4.2 多模型与设置

| 步骤 | 内容 | 负责模块 | 验收标准 |
|------|------|----------|----------|
| 4.2.1 | 设置页/后端配置支持区分：写作模型、摘要模型、审稿模型（及是否启用审稿） | backend config + settings API，frontend SettingsPage | 可分别配置并生效 |
| 4.2.2 | 文档中说明各模型推荐用途（如写作用 A、摘要用 B 以节省成本） | README 或 DEV_DOC | 有明确指引 |

### 4.3 体验与健壮性

| 步骤 | 内容 | 负责模块 | 验收标准 |
|------|------|----------|----------|
| 4.3.1 | 流式生成/保存失败时统一错误提示（toast/弹窗），并带「重试」或引导 | frontend | 用户能识别失败并重试 |
| 4.3.2 | 写作页窄屏适配：小屏下左侧 Context、右侧 Assistant 可收起为 Tab 或抽屉，中间编辑器尽量全宽 | frontend `WriterPage.vue` | 移动端/小屏可用 |
| 4.3.3 | SSE 长连接：确认 Nginx/反向代理与前端超时配置合理，避免长场景生成时被中断 | 部署与前端 | 长文生成不断连 |

### 4.4 测试与 CI

| 步骤 | 内容 | 负责模块 | 验收标准 |
|------|------|----------|----------|
| 4.4.1 | 为场景生成 SSE、RAG 按 novel 过滤补充单元/集成测试 | backend tests | CI 通过且覆盖关键路径 |
| 4.4.2 | 可选：关键用户路径 E2E（如：创建小说 → 生成大纲 → 进入写作页 → 生成一段） | e2e 目录或现有框架 | 主流程可自动回归 |

---

## 执行建议

- **阶段一**：1–2 天内完成，不依赖新功能，可立即提升稳定性和可维护性。
- **阶段二**：在阶段一之后进行，约 1–2 天，明显改善第一印象和功能可发现性。
- **阶段三**：可与阶段二部分并行，按「场景 Chat → 导出说明 → 审稿默认值」顺序做。
- **阶段四**：按需排期，版本历史和多模型对重度用户价值高，可拆成多次迭代。

每完成一个步骤，可在本文档中打勾（如将 `| 1.1.1 |` 改为 `| 1.1.1 ✅ |`）或在本仓库的 Project/Issues 中建立对应任务并关联此文档，便于跟踪进度。
